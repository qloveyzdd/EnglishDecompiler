<threat_model>
## Trust Boundaries
- boundary: raw user sentence -> parser core
  data: arbitrary technical-English sentence text
- boundary: parser output -> future demo and IR layers
  data: ordered spans, summary fields, fallback labels

## Threat Register
- threat_id: T-02-01
  category: Tampering
  component: parser output contract
  disposition: mitigate
  mitigation_plan: Define the exact public TypeScript contract in `src/parser/types.ts`, re-export it from `src/index.ts`, and keep the parse path pure so later phases read one stable structure.
- threat_id: T-02-02
  category: Integrity
  component: token role labeling
  disposition: mitigate
  mitigation_plan: Keep vocabulary in explicit lookup tables, annotate every span with a `rule` string such as `action:lexicon`, `relation:sequence`, or `fallback:unknown`, and never upgrade an unknown token into a known role without a matching rule.
- threat_id: T-02-03
  category: Denial of Service
  component: parser implementation
  disposition: mitigate
  mitigation_plan: Keep parsing synchronous and linear over one sentence only, with no regex backtracking tricks, no recursion, and no external I/O inside `parseSentence`.
</threat_model>
